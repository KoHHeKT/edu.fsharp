# Проект «Язык фильтров»

Проект относится к треку C «DSL, парсер и интерпретатор» из [общего плана](./plan02.md). Цель — создать типизированный язык запросов для фильтрации коллекции заявок или заказов:

```text
status:open and amount>=1000
customer:"Иван Петров" or tag:priority
created>=2026-01-01 and not status:cancelled
```

План начинается на **3-й неделе**, когда в общем курсе появляются discriminated unions, recursive types и pattern matching. Нагрузка на проект — **3–4 часа в неделю**, всего около 35–40 часов.

## Что должно получиться

Готовое приложение принимает выражение фильтра, разбирает его в AST, проверяет типы полей и значений, применяет к данным и возвращает результат либо диагностическую ошибку с позицией.

Минимальная цепочка:

```text
Text
-> Tokens
-> Untyped AST
-> Validated expression
-> Predicate
-> Filtered records
```

Поддерживаемые конструкции MVP:

- поля `status`, `amount`, `customer`, `created`, `tag`;
- операторы `:`, `=`, `!=`, `>`, `>=`, `<`, `<=`;
- логические операторы `and`, `or`, `not`;
- скобки и приоритет операторов;
- строки в кавычках, числа, даты и enum-подобные значения;
- понятные ошибки синтаксиса и типов.

## Архитектурные ограничения

- AST описан discriminated unions, а не иерархией классов.
- Парсер не знает структуру исходных записей.
- Семантическая проверка отделена от синтаксического разбора.
- Интерпретатор работает как чистая функция.
- Ожидаемые ошибки представлены типами, а не исключениями.
- CLI или HTTP API остаётся тонкой оболочкой.
- FParsec разрешается подключить после самостоятельной реализации базовых комбинаторов.

## Неделя 3. Грамматика и модель AST

**Связь с общим планом:** алгебраические типы, recursive types, pattern matching.

**Задачи**

1. Записать грамматику MVP в EBNF или близкой форме.
2. Определить типы `Field`, `ComparisonOperator`, `Literal` и рекурсивный `Expression`.
3. Определить records, которые будут фильтроваться.
4. Подготовить 15 корректных и 15 ошибочных примеров выражений.
5. Реализовать ручное создание AST и простой evaluator без парсера.

Пример целевой модели:

```fsharp
type Expression =
    | Comparison of Field * ComparisonOperator * Literal
    | And of Expression * Expression
    | Or of Expression * Expression
    | Not of Expression
```

**Результат недели**

- `grammar.md`;
- доменные типы и AST;
- чистая функция вычисления вручную созданного выражения;
- unit-тесты на все варианты AST.

**Критерий готовности**

Новый case в `Expression` вызывает предупреждения во всех незавершённых pattern matches.

## Неделя 4. Результаты, ошибки и базовые комбинаторы

**Связь с общим планом:** `Result`, `map`, `bind`, Railway Oriented Programming.

**Задачи**

1. Определить `SourcePosition`, `ParseError` и `ValidationError`.
2. Представить состояние парсера типом `ParserState`.
3. Реализовать базовые parser combinators: символ, строка, whitespace, последовательность, выбор и повторение.
4. Собрать парсер идентификатора, числа и строки в кавычках.
5. Не использовать исключения для обычной ошибки разбора.

**Результат недели**

- библиотека базовых комбинаторов;
- парсеры трёх видов литералов;
- ошибки с позицией и ожидаемой конструкцией;
- тесты на успех, ошибку и оставшийся ввод.

**Критерий готовности**

Для ввода `amount>=abc` результат содержит позицию и ожидаемый тип, а не `FormatException`.

## Неделя 5. Модули и публичный API

**Связь с общим планом:** организация F#-кода, signatures, active patterns, interop.

**Задачи**

1. Разделить код на `Syntax`, `Parsing`, `Validation`, `Evaluation` и `PublicApi`.
2. Скрыть внутренние конструкторы состояния парсера.
3. Определить минимальный API:

```fsharp
parse : string -> Result<Expression, ParseError>
validate : Schema -> Expression -> Result<TypedExpression, ValidationError list>
evaluate : TypedExpression -> Record -> bool
```

4. Добавить `.fsi` для публичного модуля или проверить публичные сигнатуры вручную.
5. Сделать C#-совместимый фасад `FilterEngine.TryCompile`.

**Результат недели**

- устойчивые границы модулей;
- скрытая внутренняя реализация;
- минимальная демонстрация вызова из C#.

**Критерий готовности**

Потребитель не может создать некорректный `TypedExpression` напрямую.

## Неделя 6. Асинхронный источник данных

**Связь с общим планом:** `task {}`, отмена, параллельная композиция.

**Задачи**

1. Добавить асинхронную загрузку записей из JSON-файла или HTTP endpoint.
2. Передавать загрузчик как функцию-зависимость.
3. Поддержать `CancellationToken`.
4. Разделить ошибки загрузки, разбора фильтра и семантической проверки.
5. При нескольких источниках загружать их параллельно через `and!` или `Task.WhenAll`.

**Результат недели**

- workflow `load -> compile filter -> apply`;
- отменяемая операция;
- тест workflow с подменённым загрузчиком.

**Критерий готовности**

Интерпретатор не зависит от `Task`, HTTP, файловой системы и сериализации.

## Неделя 7. Декодирование данных и схема полей

**Связь с общим планом:** внешние форматы, DTO и доменные типы.

**Задачи**

1. Отделить JSON DTO от доменной записи.
2. Валидировать обязательные поля входных данных.
3. Описать `Schema`, связывающую имя поля с его типом.
4. Добавить семантические ошибки: неизвестное поле, неподдерживаемый оператор, несовместимый литерал.
5. Превратить untyped AST в typed expression.

**Результат недели**

- явная граница декодирования;
- типизированное представление проверенного фильтра;
- накопление независимых ошибок схемы, где это возможно.

**Критерий готовности**

`amount:"open"` и `unknown:42` проходят синтаксический разбор, но возвращают разные семантические ошибки.

## Неделя 8. Property-based testing и pretty-printer

**Связь с общим планом:** FsCheck, generators, shrinking, round trip.

**Задачи**

1. Написать генератор корректных AST ограниченной глубины.
2. Реализовать pretty-printer с минимально необходимыми скобками.
3. Проверить свойство `parse (print ast) = ast`.
4. Проверить идемпотентность форматирования.
5. Проверить законы логики на случайных записях: double negation, `x and true`, `x or false`.
6. Превращать найденные контрпримеры в отдельные regression tests.

**Результат недели**

- не менее пяти property tests;
- shrinking до читаемых выражений;
- стабильный pretty-printer.

**Критерий готовности**

Несколько сотен случайных AST проходят round-trip без изменения смысла.

## Что проект демонстрирует

- recursive discriminated unions естественно описывают AST;
- parser combinators позволяют собирать сложное поведение из малых функций;
- pattern matching делает интерпретатор прямым и проверяемым компилятором;
- типизированная семантическая стадия исключает несовместимые операции до выполнения;
- добавление оператора показывает реальные различия между union-based и class hierarchy-based дизайном.
