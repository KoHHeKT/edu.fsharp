# Проект «Обработка заказов интернет-магазина»

Проект относится к треку A «Доменный workflow» из [общего плана](./plan02.md). Цель — реализовать типизированное ядро обработки заказа от черновика до оплаты, отгрузки, отмены или отказа.

План начинается на **3-й неделе**, когда изучаются records, discriminated unions и принцип «make illegal states unrepresentable». Нагрузка — **3–4 часа в неделю**, всего около 38–42 часов.

## Границы проекта

Основные use cases:

- создать и изменить черновик;
- разместить заказ;
- подтвердить или отклонить оплату;
- отгрузить оплаченный заказ;
- отменить заказ по допустимому правилу;
- получить историю событий заказа.

Базовый жизненный цикл:

```text
Draft
  -> Placed
      -> PaymentFailed
      -> Paid
          -> Shipped
      -> Cancelled
```

Необязательные расширения:

- резервирование товара;
- промокоды;
- частичная отгрузка;
- возврат;
- разные способы доставки;
- расчёт налогов.

Расширение выбирайте только после завершения базовых use cases.

## Доменные инварианты

Минимальный набор:

1. Заказ содержит хотя бы одну строку.
2. Количество товара положительно.
3. Цена и итоговая сумма не отрицательны.
4. Разместить можно только корректный черновик.
5. Оплатить можно только размещённый заказ.
6. Отгрузить можно только оплаченный заказ.
7. Отменённый или отгруженный заказ нельзя изменять.
8. Сумма заказа равна сумме строк с учётом допустимых скидок.

## Архитектурные ограничения

- Состояния заказа представлены разными union cases, а не `Status` плюс nullable-поля.
- Доменные значения создаются smart constructors.
- Ожидаемые отказы возвращаются через `Result`.
- Часы, идентификаторы, каталог, платёжный шлюз и хранилище передаются как функции.
- Домен не зависит от ASP.NET Core, JSON, SQL и ORM.
- DTO отделены от доменных типов.

## Неделя 3. Модель заказа и допустимые состояния

**Связь с общим планом:** records, discriminated unions, single-case unions, pattern matching.

**Задачи**

1. Составить словарь домена.
2. Определить `OrderId`, `ProductCode`, `Quantity`, `Money`, `EmailAddress`.
3. Смоделировать `DraftOrder`, `PlacedOrder`, `PaidOrder`, `ShippedOrder`.
4. Представить общий жизненный цикл union-типом `Order`.
5. Записать таблицу допустимых команд для каждого состояния.
6. Реализовать расчёт суммы заказа как чистую функцию.

**Результат недели**

- `domain-glossary.md`;
- модель состояний;
- smart constructors основных значений;
- таблица переходов;
- unit-тесты расчётов.

**Критерий готовности**

Нельзя создать `ShippedOrder` без данных об оплате и отгрузке.

## Неделя 4. Валидация и workflow размещения

**Связь с общим планом:** `Result`, накопительная валидация, ROP.

**Задачи**

1. Определить входной DTO для команды размещения.
2. Реализовать независимую валидацию клиента, адреса и строк заказа.
3. Накопить все ошибки формы до обращения к внешним системам.
4. Проверять существование товара и актуальную цену через функции-зависимости.
5. Собрать `placeOrder` как pipeline.
6. Разделить validation, business и infrastructure errors.

Пример сигнатуры:

```fsharp
placeOrder :
    CheckProductCode
    -> GetProductPrice
    -> PlaceOrderCommand
    -> Result<PlacedOrder * OrderEvent list, PlaceOrderError>
```

**Результат недели**

- чистый workflow размещения;
- типизированные ошибки;
- тесты happy path и всех ветвей отказа.

**Критерий готовности**

Все бизнес-правила размещения проверяются без базы данных и mocking framework.

## Неделя 5. Команды, события и публичный API

**Связь с общим планом:** модули, signatures, сокрытие конструкторов, C# interop.

**Задачи**

1. Ввести `OrderCommand` и `OrderEvent`.
2. Разделить код на `DomainTypes`, `Validation`, `Workflows`, `PublicApi`.
3. Скрыть конструкторы проверенных типов.
4. Определить публичные функции для `place`, `pay`, `ship`, `cancel`.
5. Создать C#-совместимый фасад для размещения заказа.
6. Проверить, какие F#-типы стоит скрыть за DTO на interop-границе.

**Результат недели**

- минимальный публичный API;
- команды и события;
- вызов `PlaceOrder` из C#.

**Критерий готовности**

C#-потребитель не может обойти smart constructors и собрать внутреннее состояние заказа.

## Неделя 6. Асинхронная orchestration

**Связь с общим планом:** `task {}`, параллельные зависимости, отмена.

**Задачи**

1. Добавить асинхронные проверки каталога и клиента.
2. Выполнять независимые проверки параллельно.
3. Поддержать `CancellationToken`.
4. Добавить timeout для платёжного шлюза.
5. Отделить решение домена от фактического вызова платёжного API.
6. Протестировать orchestration функциями-заглушками.

**Результат недели**

- отменяемые application workflows;
- параллельная проверка независимых данных;
- явная классификация внешних ошибок.

**Критерий готовности**

Timeout платёжного шлюза не превращается автоматически в доменный отказ оплаты.

## Неделя 7. DTO, сериализация и внешние контракты

**Связь с общим планом:** внешние данные, декодирование, transport и domain types.

**Задачи**

1. Создать JSON DTO для команд и ответов.
2. Реализовать явное преобразование DTO в доменные значения.
3. Не добавлять `CLIMutable` в доменное ядро ради сериализатора.
4. Описать versioned DTO для событий.
5. Проверить обработку отсутствующих и неизвестных полей.
6. Подготовить fixtures входных заказов.

**Результат недели**

- transport layer;
- сериализация команд и событий;
- тесты декодирования.

**Критерий готовности**

Некорректный JSON не создаёт частично валидный доменный заказ.

## Неделя 8. Property-based testing инвариантов

**Связь с общим планом:** FsCheck, generators и shrinking.

**Задачи**

1. Создать генераторы корректных строк и заказов.
2. Проверить, что итог равен сумме строк.
3. Проверить, что положительные quantities сохраняются после допустимых операций.
4. Проверить запрет отгрузки неоплаченного заказа.
5. Проверить запрет изменения финального состояния.
6. Проверить, что последовательность допустимых команд не нарушает инварианты.
7. Добавить regression tests для найденных контрпримеров.

**Результат недели**

- не менее пяти property tests;
- генератор последовательностей команд;
- unit-тесты конкретных бизнес-примеров.

**Критерий готовности**

Случайная последовательность команд никогда не создаёт противоречивое состояние.

## Неделя 9. Functional core и imperative shell

**Связь с общим планом:** workflow architecture, commands, events, persistence ignorance.

**Задачи**

1. Оформить чистую функцию обработки команды:

```fsharp
decide : Order -> OrderCommand -> Result<OrderEvent list, DomainError>
evolve : Order -> OrderEvent -> Order
```

2. Восстанавливать состояние свёрткой событий либо оставить обычный repository с audit trail.
3. Реализовать in-memory repository.
4. Передать часы и генератор идентификаторов как зависимости.
5. Нарисовать схему functional core/imperative shell.
6. Проверить идемпотентность повторной команды оплаты, если она предусмотрена контрактом.

**Результат недели**

- чистые `decide` и `evolve`;
- in-memory persistence;
- audit trail;
- архитектурная схема.

**Критерий готовности**

Замена способа хранения не требует изменения доменных переходов.

## Неделя 10. HTTP API и persistence

**Связь с общим планом:** ASP.NET Core, сериализация, базы данных.

**Задачи**

1. Добавить endpoints размещения, оплаты, отгрузки и чтения заказа.
2. Отобразить доменные ошибки в стабильные HTTP responses.
3. Реализовать persistence через SQLite, PostgreSQL или файловое хранилище.
4. Добавить optimistic concurrency либо проверку версии.
5. Написать integration tests на три сквозных сценария.
6. Оставить endpoints тонкими: decode, call workflow, map result.

**Результат недели**

- работающий HTTP API;
- persistence;
- integration tests;
- обработка concurrent update.

**Критерий готовности**

Ни одно бизнес-правило не реализовано только в controller или SQL.

## Неделя 11. Interop, benchmark и C#-сравнение

**Связь с общим планом:** .NET interop и производительность.

**Задачи**

1. Реализовать use case размещения заказа на C#.
2. Сравнить F# unions с C# class hierarchy, enum/status и nullable fields.
3. Добавить новый вариант `PaymentFailed` или `PartiallyShipped`.
4. Зафиксировать места, которые потребовали изменения в обоих решениях.
5. Измерить обработку большой корзины и replay событий, если он используется.
6. Оптимизировать только подтверждённое узкое место.

**Результат недели**

- C#-реализация центрального сценария;
- change-impact comparison;
- BenchmarkDotNet-отчёт при наличии значимого участка.

**Критерий готовности**

Сравнение показывает не длину кода, а допустимые состояния, exhaustiveness, тестируемость и стоимость изменения.

**Связь с общим планом:** стиль, Fantomas, документация и итоговая проверка.

**Задачи**

3. Подготовить README с жизненным циклом заказа и командами запуска.

5. Подготовить диаграмму states/commands/events.
6. Провести демонстрацию happy path и минимум трёх запрещённых операций.
7. Проверить чистый checkout через `dotnet build` и `dotnet test`.

**Итоговый результат**

- use cases `PlaceOrder`, `PayOrder`, `ShipOrder`, `CancelOrder`;
- типизированные состояния и ошибки;
- audit trail;
- HTTP API и persistence;
- unit-, property- и integration tests;
- C#-фасад и сравнительный сценарий.

## Что проект демонстрирует

- типы исключают невозможные сочетания статуса и данных;
- discriminated unions и exhaustive matching делают переходы явными;
- чистые workflows тестируются без инфраструктуры;
- функции-зависимости заменяют значительную часть DI- и mocking-механики;
- изменение жизненного цикла заказа показывает, как компилятор помогает найти все затронутые места.
