# Проект «Анализ истории Git»

Проект относится к треку B «Типизированный ETL и аналитический pipeline» из [общего плана](./plan02.md). Цель — создать воспроизводимый анализатор Git-репозиториев, который извлекает историю коммитов, валидирует данные, рассчитывает метрики и формирует Markdown- или HTML-отчёт.

План начинается на **2-й неделе**, одновременно с изучением коллекций. Нагрузка — **3–4 часа в неделю**, всего около 40–45 часов.

## Исследовательские вопросы

MVP должен отвечать минимум на пять вопросов:

- кто и когда вносит изменения;
- как меняется активность по неделям и месяцам;
- какие файлы и каталоги меняются чаще всего;
- какие файлы часто изменяются вместе;
- каков размер и типичный жизненный цикл коммита;
- где наблюдается концентрация владения кодом;
- какие периоды или участки истории содержат аномалии.

Не превращайте метрики в оценку продуктивности разработчиков. Количество коммитов и строк не измеряет ценность работы.

## Источники данных

Основной источник — вывод Git CLI в машинно-читаемом формате. Дополнительные источники:

- `git log` с custom pretty format;
- `git numstat`;
- `git shortlog`;
- метаданные репозитория;
- необязательный JSON GitHub API для pull requests или contributors.

Для MVP достаточно одного локального репозитория. Итоговая версия должна поддерживать сравнение 2–3 репозиториев либо нескольких периодов одного репозитория.

## Целевой pipeline

```text
Git process output
-> Raw records
-> Decoded commits and changes
-> Normalized identities
-> Validated history
-> Aggregations
-> Findings
-> Markdown/HTML report
```

## Неделя 2. Исследование данных и первые агрегаты

**Связь с общим планом:** коллекции, `map`, `choose`, `fold`, `groupBy`, `Map`, `Set`.

**Задачи**

1. Создать `.fsx` и получить небольшой образец истории через `git log`.
2. Сохранить fixture с 20–50 коммитами, чтобы эксперименты были воспроизводимыми.
3. Описать простые records `RawCommit` и `RawFileChange`.
4. Рассчитать число коммитов по автору и месяцу.
5. Сравнить реализацию через специализированные функции коллекций и `fold`.
6. Зафиксировать ограничения выбранного формата вывода Git.

**Результат недели**

- исследовательский скрипт;
- первая таблица агрегатов;
- fixture без зависимости тестов от текущего состояния репозитория.

**Критерий готовности**

Один и тот же fixture всегда даёт одинаковый результат независимо от рабочей директории.

## Неделя 3. Типизированная модель истории

**Связь с общим планом:** records, unions, single-case unions, `option`.

**Задачи**

1. Создать типы `CommitHash`, `AuthorEmail`, `RepositoryPath`, `CommitTime`.
2. Представить вид изменения union-типом `Added | Modified | Deleted | Renamed`.
3. Разделить raw, validated и normalized records.
4. Описать merge-коммит явно, а не булевым флагом.
5. Определить правила нормализации автора и пути.
6. Смоделировать неизвестные или неполные значения через `option`.

**Результат недели**

- доменная модель истории Git;
- smart constructors;
- схема стадий pipeline.

**Критерий готовности**

Хеш коммита, email и путь нельзя случайно перепутать как обычные строки.

## Неделя 4. Декодирование и отчёт об ошибках

**Связь с общим планом:** `Result`, композиция ошибок, чистые функции.

**Задачи**

1. Выбрать устойчивый разделитель и формат вывода Git.
2. Реализовать чистый decoder raw text в records.
3. Ввести `DecodeError` с номером записи и проблемным полем.
4. Не терять повреждённые строки: формировать `DataQualityReport`.
5. Разделить fatal errors запуска Git и recoverable errors отдельных записей.
6. Сравнить fail-fast и накопительную обработку ошибок.

**Результат недели**

- decoder;
- отчёт о качестве входных данных;
- unit-тесты на корректные и повреждённые fixtures.

**Критерий готовности**

Одна повреждённая запись не скрывает остальные ошибки и не уничтожает корректную часть отчёта.

## Неделя 5. Структура решения и публичный API

**Связь с общим планом:** модули, signatures, interop.

**Задачи**

1. Разделить решение на `GitAccess`, `Decoding`, `Domain`, `Analysis`, `Reporting`.
2. Скрыть raw constructors за функциями декодирования.
3. Определить API анализа:

```fsharp
analyze : AnalysisOptions -> Commit list -> AnalysisResult
```

4. Описать `AnalysisOptions`: период, ветка, исключённые пути, обработка merge commits.
5. Сделать маленький C#-фасад для запуска анализа.

**Результат недели**

- направленный граф зависимостей;
- стабильная функция анализа;
- вызов F#-библиотеки из C#.

**Критерий готовности**

Чистый анализ не запускает процессы и не читает файловую систему.

## Неделя 6. Асинхронный сбор нескольких историй

**Связь с общим планом:** `task {}`, параллельная композиция, отмена.

**Задачи**

1. Реализовать безопасный запуск Git через `System.Diagnostics.Process`.
2. Передавать executable runner как функцию-зависимость.
3. Поддержать timeout, `CancellationToken`, exit code и stderr.
4. Параллельно собирать историю 2–3 репозиториев или периодов.
5. Ограничить параллелизм.
6. Разделить ошибки «Git не запущен», «репозиторий недоступен» и «данные не декодированы».

**Результат недели**

- отменяемый collector;
- параллельный сбор;
- тесты orchestration без запуска реального Git.

**Критерий готовности**

Сбой одного репозитория представлен отдельно и не уничтожает успешные результаты остальных.

## Неделя 7. Обогащение и дополнительные источники

**Связь с общим планом:** внешние данные, JSON, transport и domain types.

**Задачи**

1. Добавить конфигурационный JSON с aliases авторов и исключёнными путями.
2. Валидировать конфигурацию и преобразовать её в доменные типы.
3. Нормализовать разные email одного автора.
4. При желании подключить GitHub API для metadata репозитория.
5. Подготовить промежуточный экспорт normalized history в JSON или CSV.
6. Проверить round trip экспорта.

**Результат недели**

- enrichment stage;
- конфигурация aliases;
- минимум два формата данных в pipeline.

**Критерий готовности**

Изменение transport DTO не требует изменения аналитических функций.

## Неделя 8. Property-based testing агрегатов

**Связь с общим планом:** FsCheck и инварианты.

**Задачи**

1. Написать генераторы коммитов, авторов и file changes.
2. Проверить, что сумма коммитов по группам равна числу входных коммитов.
3. Проверить независимость результата от порядка входа для commutative aggregations.
4. Проверить идемпотентность нормализации путей и авторов.
5. Проверить, что фильтрация периода не возвращает данные вне диапазона.
6. Проверить round trip промежуточного формата.

**Результат недели**

- не менее пяти property tests;
- генераторы реалистичных историй;
- regression tests для найденных контрпримеров.

**Критерий готовности**

Агрегации проверены на сотнях случайных историй, а не только на одном репозитории.

## Неделя 9. Аналитическое ядро

**Связь с общим планом:** functional core, imperative shell.

**Задачи**

1. Оформить метрики как независимые чистые функции.
2. Реализовать активность по времени, churn по путям и ownership concentration.
3. Реализовать co-change analysis через пары файлов.
4. Ввести типизированные findings вместо готовых строк отчёта.
5. Отделить расчёт метрик от их интерпретации и форматирования.
6. Нарисовать схему pipeline.

**Результат недели**

- библиотека аналитических функций;
- `AnalysisResult`;
- минимум пять findings;
- отсутствие I/O в аналитическом ядре.

**Критерий готовности**

Любую метрику можно протестировать передачей обычного списка коммитов.

## Неделя 10. CLI и воспроизводимый отчёт

**Связь с общим планом:** CLI, сериализация, конфигурация.

**Задачи**

1. Создать CLI с параметрами репозитория, периода, ветки и формата.
2. Сформировать Markdown-отчёт с таблицами и выводами.
3. Добавить HTML-графики через Plotly.NET либо сохранить данные для визуализации.
4. Указать методику и ограничения каждой метрики.
5. Добавить integration test на fixture-репозиторий.
6. Исключить оценочные формулировки о разработчиках без достаточных данных.

**Результат недели**

- команда `analyze`;
- Markdown- или HTML-отчёт;
- integration test;
- описание методики.

**Критерий готовности**

Отчёт полностью воспроизводится одной командой из зафиксированного состояния репозитория.

## Неделя 11. Benchmark и C#-сравнение

**Связь с общим планом:** производительность, коллекции и interop.

**Задачи**

1. Создать synthetic history на 10 000–100 000 коммитов.
2. Измерить декодирование, co-change analysis и группировки.
3. Сравнить `list`, `array`, `seq`, `Map` и `Dictionary` на реальном узком месте.
4. Оптимизировать только подтверждённый hotspot.
5. Реализовать одну метрику на C# LINQ.
6. Сравнить типы промежуточных стадий, тестируемость, allocations и скорость.

**Результат недели**

- BenchmarkDotNet-отчёт;
- обоснованный выбор коллекций;
- C#-реализация одной аналитической цепочки;
- сравнительная таблица.

**Критерий готовности**

В выводах разделены удобство моделирования, читаемость и фактическая производительность.

## Неделя 12. Выпуск

**Связь с общим планом:** стиль, предупреждения и документация.

**Задачи**

1. Применить Fantomas и устранить предупреждения.
2. Проверить поведение на Windows- и Unix-путях в fixtures.
3. Подготовить README с примерами команд и образцом отчёта.
4. Добавить ADR «Почему аналитический pipeline реализован на F#».
5. Зафиксировать ограничения: merge commits, renames, binary files, shallow clones.
6. Проверить `dotnet build`, `dotnet test` и генерацию отчёта из чистого checkout.

**Итоговый результат**

- анализ одного или нескольких Git-репозиториев;
- минимум пять метрик;
- отчёт о качестве данных;
- Markdown/HTML-отчёт;
- unit-, property- и integration tests;
- benchmark и сравнение с LINQ.

## Что проект демонстрирует

- F# удобен для REPL-driven исследования и последующего переноса pipeline в production-код;
- типы фиксируют смысл каждой стадии ETL;
- композиция коллекционных функций делает преобразования локальными и проверяемыми;
- чистые агрегаты легко тестировать случайными историями;
- локальная оптимизация позволяет сохранить функциональную архитектуру без отказа от производительности.
