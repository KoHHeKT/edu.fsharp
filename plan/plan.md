# Самостоятельное изучение F# за три месяца

План рассчитан на опытного .NET/C#-разработчика и занимает **12 недель по 8–10 часов**. Главная цель — не выучить альтернативный синтаксис .NET, а освоить функциональный способ проектирования: моделировать данные типами, собирать поведение из функций, отделять вычисления от эффектов и переносить часть ошибок из runtime в compile time.

Понедельный план намеренно не содержит учебного проекта. Небольшие упражнения нужны для освоения конкретной темы. Полноценные варианты проектов собраны отдельно в конце документа: можно выбрать один трек и проходить его параллельно учебной программе либо начать после восьмой недели.

## Ожидаемый результат

Через три месяца вы должны уметь:

- писать идиоматичный F#, а не C# без фигурных скобок;
- использовать type inference, currying, partial application, composition и pipelines;
- моделировать предметную область через records, discriminated unions и single-case unions;
- делать недопустимые состояния непредставимыми;
- применять `option`, `Result` и computation expressions для явного управления отсутствующими значениями и ошибками;
- отделять чистое функциональное ядро от инфраструктуры;
- выбирать между `list`, `array`, `seq`, `Map` и `Set`;
- писать асинхронные workflow на `task {}` и понимать назначение `async {}`;
- применять example-based и property-based testing;
- использовать библиотеки .NET из F# и предоставлять удобный F# API для C#;
- аргументированно объяснять, где F# даёт преимущество, а где C# остаётся практичнее.

## Режим обучения

Рекомендуемый недельный ритм:

| Работа | Время |
|---|---:|
| Чтение книг | 2,5–3 ч |
| Документация, статьи, видео | 1–1,5 ч |
| Упражнения в `.fsx` и небольших лабораторных решениях | 3–4 ч |
| Рефакторинг и журнал обучения | 1 ч |

Рабочие правила:

1. Исследуйте незнакомую конструкцию в F# Interactive или `.fsx`, затем переносите устойчивое решение в `.fs`.
2. Перед реализацией записывайте типы и сигнатуры функций.
3. По умолчанию используйте неизменяемые значения. Мутабельность оставляйте локальной и обоснованной измерениями.
4. Не переносите автоматически классы, интерфейсы, DI-контейнеры и паттерны C#. Сначала рассмотрите функции, records, unions и модули.
5. Для каждой крупной темы делайте маленькую C#-версию и сравнивайте не число строк, а число возможных ошибок, связность, тестируемость и стоимость изменения.
6. Ведите `learning-log.md`: новые идеи, ошибки мышления, полезные сигнатуры, сравнение с C# и вопросы для повторения.
7. Все упражнения храните в Git. В конце недели оставляйте короткую ретроспективу и один отрефакторенный пример.

# Проектные треки

Выберите **один основной трек**. Проект должен быть достаточно небольшим, чтобы закончить его за 35–50 часов, но достаточно содержательным, чтобы показать типы, композицию, чистое ядро, работу с эффектами и тестирование свойств.

Общие требования для любого трека:

- отдельные проекты `Domain`, `Application`, `Infrastructure` и `Tests`, если это оправдано размером;
- минимум пять осмысленных доменных типов;
- типизированные ожидаемые ошибки;
- не менее пяти unit tests и пяти property-based tests;
- хотя бы одна асинхронная инфраструктурная операция;
- тонкая CLI- или HTTP-оболочка;
- один центральный сценарий, реализованный также на C# для сравнения;
- README, схема типов или pipeline и ADR «Почему здесь F#»;
- честное описание не только преимуществ, но и неудобств F#.

## Трек A. Доменный workflow

### Варианты

- обработка заказов интернет-магазина;
- бронирование переговорных, билетов или оборудования;
- биллинг подписок;
- согласование документов;
- страховой claim;
- складские перемещения.

### Ядро проекта

Опишите команды, состояния и события. Например:

```text
Draft -> Placed -> Paid -> Shipped
   \         \         \
    \         -> Cancelled
     -> Rejected
```

Переходы должны выполняться чистой функцией, а недопустимые операции возвращать конкретный тип ошибки. Валидация входных данных создаёт проверенные доменные значения. Хранилище, часы, генератор идентификаторов и уведомления передаются как функции.

### Обязательные элементы

- 6–10 single-case unions;
- discriminated union для состояний;
- команды и события;
- накопительная валидация;
- audit trail;
- минимум пять инвариантов;
- сериализация через отдельные DTO;
- C#-фасад для одного use case.

### Что показывает F#

- типы работают как исполняемая спецификация;
- exhaustive matching заменяет распределённые `if` и `switch`;
- невозможно получить заказ с противоречивыми флагами;
- чистые бизнес-правила тестируются без mocking framework;
- новый вариант состояния обнаруживается компилятором во всех значимых местах.

### Этапы

1. Словарь домена, команды, состояния и инварианты.
2. Чистые переходы и валидация.
3. Property-based tests.
4. In-memory-инфраструктура.
5. HTTP/CLI, persistence и C#-сравнение.

## Трек B. Типизированный ETL и аналитический pipeline

### Варианты

- анализ банковских операций;
- агрегация телеметрии;
- анализ истории Git;
- контроль качества открытых данных;
- сбор публичных цен;
- обработка логов нескольких сервисов.

### Ядро проекта

Соберите данные минимум из трёх источников или форматов. Каждая стадия pipeline должна принимать и возвращать понятный тип:

```text
Raw input
-> Decoded row
-> Validated record
-> Enriched record
-> Aggregation
-> Report
```

Ошибочные строки не теряются: формируется отчёт о качестве данных. Независимые источники загружаются параллельно. Результат визуализируется или экспортируется в воспроизводимый отчёт.

### Обязательные элементы

- исследовательский `.fsx` или notebook;
- FSharp.Data либо явные декодеры;
- `Result` для ошибок строк;
- `Map`, `Set` и осмысленный выбор коллекций;
- параллельная загрузка;
- Plotly.NET, Markdown- или HTML-отчёт;
- benchmark одной стадии;
- property tests для агрегаций и round trip.

### Что показывает F#

- декларативные преобразования данных;
- типизированные границы между стадиями;
- REPL-driven development;
- естественная композиция небольших функций;
- локальная оптимизация горячей стадии без мутабельной архитектуры целиком.

### Этапы

1. Исследование источников и модель сырых данных.
2. Декодирование, очистка и отчёт об ошибках.
3. Обогащение и агрегации.
4. Параллельная загрузка и визуализация.
5. Benchmark и сравнение с LINQ-версией.

## Трек C. DSL, парсер и интерпретатор

### Варианты

- язык правил скидок;
- фильтры `status:open amount>100`;
- мини-язык расписаний;
- валидатор конфигурации;
- выражения для маршрутизации сообщений;
- язык расчёта комиссий.

### Ядро проекта

Опишите грамматику, AST на discriminated unions, parser combinators и чистый интерпретатор. Парсер можно собрать самостоятельно для обучения либо использовать [FParsec](https://www.quanttec.com/fparsec/).

Пример стадий:

```text
Text -> Tokens -> AST -> Validation -> Evaluation
```

Ошибки должны содержать позицию, ожидаемую конструкцию и контекст. Добавьте pretty-printer и round-trip property.

### Обязательные элементы

- формальная или полуформальная грамматика;
- 10–15 конструкций языка;
- рекурсивный AST;
- parser combinators;
- pattern matching интерпретатора;
- типизированные parse и evaluation errors;
- `parse (print ast) = ast`;
- расширение языка новым оператором.

### Что показывает F#

- алгебраические типы естественно представляют деревья;
- рекурсия и pattern matching заменяют иерархии классов и Visitor;
- сложный парсер собирается из малых комбинаторов;
- компилятор контролирует покрытие вариантов AST;
- грамматика, типы и интерпретатор остаются близки друг к другу.

### Этапы

1. Грамматика и AST.
2. Базовые parser combinators.
3. Валидация и интерпретатор.
4. Диагностика и pretty-printer.
5. Расширение языка и C#-версия через классы/Visitor.

## Трек D. Event-sourced конечный автомат

### Варианты

- банковский счёт;
- кошелёк бонусных баллов;
- торговый автомат;
- жизненный цикл заявки;
- управление доступом;
- партия пошаговой игры.

### Ядро проекта

Состояние восстанавливается свёрткой событий, а команда обрабатывается чистой функцией:

```fsharp
evolve : State -> Event -> State
decide : State -> Command -> Result<Event list, DomainError>
```

События неизменяемы и версионируются. Хранилище событий остаётся инфраструктурной деталью. Добавьте optimistic concurrency и идемпотентность команд, если они уместны.

### Обязательные элементы

- discriminated unions для команд и событий;
- `fold` для восстановления состояния;
- snapshot как необязательная оптимизация;
- optimistic concurrency;
- сериализация с версией;
- property tests инвариантов;
- replay и audit;
- проекция хотя бы одного read model.

### Что показывает F#

- состояние выводится из истории чистой свёрткой;
- commands и events образуют закрытые наборы вариантов;
- переходы легко тестировать последовательностями случайных команд;
- неизменяемые события уменьшают скрытую связанность;
- модель автомата компактнее класса с множеством меняющихся полей.

### Этапы

1. Команды, события и инварианты.
2. `decide` и `evolve`.
3. Генераторы последовательностей для FsCheck.
4. Event store и projection.
5. Конкурентное обновление и C#-сравнение.

## Трек E. Конкурентный агрегатор внешних сервисов

### Варианты

- сравнение цен;
- сбор статусов CI/CD;
- агрегатор погоды;
- мониторинг доступности API;
- сводка GitHub-репозиториев;
- обогащение карточки клиента данными нескольких сервисов.

### Ядро проекта

Получайте данные из 4–6 независимых источников. Опишите результат каждого вызова типом: успех, ожидаемый отказ, timeout, rate limit или повреждённый ответ. Поддержите ограничение параллелизма, отмену, retry только для безопасных случаев и частичный итог.

### Обязательные элементы

- `task {}` и параллельная композиция;
- `CancellationToken`;
- bounded concurrency;
- типизированная классификация ошибок;
- частичный успешный результат;
- injectable clock и HTTP-функции;
- метрики длительности;
- property tests для функции объединения результатов.

### Что показывает F#

- асинхронный workflow читается как последовательное описание;
- ошибки и частичный успех явно отражены в типах;
- чистая политика retry/merge отделена от HTTP;
- функции-зависимости позволяют тестировать сценарии без сложных моков;
- union cases лучше передают семантику, чем набор nullable-полей и исключений.

### Этапы

1. Типы ответов и политика ошибок.
2. Клиенты и декодирование.
3. Параллельная orchestration.
4. Отмена, retry и ограничение нагрузки.
5. Метрики, HTTP/CLI и сравнение с C#.

## Трек F. Функциональный UI на архитектуре MVU

### Варианты

- персональный трекер задач;
- редактор правил;
- обозреватель логов;
- менеджер расходов;
- клиент для REST API;
- визуализатор конечного автомата.

### Технологические варианты

- Fable + Elmish/Feliz для web;
- Avalonia.FuncUI для desktop;
- Elmish.WPF для интеграции с WPF.

### Ядро проекта

Состояние интерфейса хранится в неизменяемой модели, пользовательские действия представлены union-типом сообщений, а изменение состояния выполняется чистой функцией:

```fsharp
update : Msg -> Model -> Model * Cmd<Msg>
```

Сетевые и файловые операции представлены командами, а не выполняются внутри `update`.

### Обязательные элементы

- `Model`, `Msg`, `update`, `view`;
- минимум восемь вариантов сообщений;
- явные состояния `NotAsked`, `Loading`, `Loaded`, `Failed`;
- валидация формы через доменные типы;
- один асинхронный эффект;
- сохранение и восстановление состояния;
- unit- и property tests для `update`;
- обработка повторных запросов и stale responses.

### Что показывает F#

- UI становится конечным автоматом с явными состояниями;
- чистый `update` тестируется без запуска интерфейса;
- exhaustive matching не позволяет забыть новый тип сообщения;
- исчезает часть скрытого состояния, двусторонних binding и event-handler spaghetti;
- модель обновления проще воспроизводить и отлаживать, чем набор изменяемых ViewModel.

### Этапы

1. Модель состояний и сообщений.
2. Чистый `update` и тесты.
3. Представление и навигация.
4. Эффекты и persistence.
5. Диагностика, race conditions и сравнение с MVVM на C#.
